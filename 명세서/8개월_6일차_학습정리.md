## 1. 개요

이번 수업의 핵심은 **HeightMap 기반 Terrain 시스템**과 **Camera 클래스의 뼈대**를 구축하는 것입니다.

| 구분 | 변경 내용 |
|------|-----------|
| 신규 Engine 클래스 | `CCamera` (abstract, 뼈대만) |
| 신규 HLSL | `Shader_VtxNorTex.hlsl` (Normal 포함 정점 셰이더) |
| 신규 구조체 | `VTXNORTEX` (Position + Normal + Texcoord) |
| 전면 재작성 | `CVIBuffer_Terrain` (HeightMap BMP → 동적 지형 메시) |
| 구조체 개선 | `VTXTEX`에 `Elements[]`/`iNumElements` 정적 멤버 추가 |
| Loader 변경 | GamePlay 레벨에 Shader_VtxNorTex + VIBuffer_Terrain 프로토타입 등록 |
| Terrain 변경 | VtxNorTex 셰이더 사용, 임시 하드코딩 View/Proj 행렬 바인딩 |
| MainApp 변경 | Shader 생성 시 구조체 정적 멤버 활용 |

### A 프로젝트와 B 수업 코드의 구조적 차이 (참고)

A 프로젝트에서는 `CTransform`을 abstract 베이스 + `CTransform_3D`/`CTransform_2D`로 분리했으나, B 수업에서는 단일 `CTransform(final)`을 유지합니다.
이 차이로 인해 `Engine_Enum.h`의 `TRANSFORMTYPE`, `GAMEOBJECT_DESC`의 `eTransformType`, `UIOBJECT_DESC` 생성자 등이 다릅니다.
**이 문서에서는 Transform 분리 관련 차이는 다루지 않고, 6일차 수업에서 새로 도입된 내용만 정리합니다.**

## 2. 정점 구조체 정적 멤버 추가

### 이론/개념

기존에는 Shader를 생성할 때 `D3D11_INPUT_ELEMENT_DESC` 배열을 매번 로컬에서 선언하여 전달했습니다.
6일차에서는 정점 구조체 자체에 `Elements[]`와 `iNumElements`를 **정적 멤버**로 포함시켜, 구조체와 InputLayout 정보가 항상 함께 다니도록 개선했습니다.

**장점:**
- Shader 생성 코드가 간결해짐 (`VTXTEX::Elements, VTXTEX::iNumElements`)
- 정점 포맷과 InputLayout 정의가 한 곳에 모여 있어 불일치 방지
- 새 정점 포맷 추가 시 구조체 하나만 정의하면 됨

### 코드 변경 (Engine_Struct.h)

**Before (A 프로젝트):**
```cpp
typedef struct tagVertexPositionTexcoord
{
    XMFLOAT3        vPosition;
    XMFLOAT2        vTexcoord;
}VTXTEX;
```

**After (B 수업):**
```cpp
typedef struct tagVertexPositionTexcoord
{
    XMFLOAT3        vPosition;
    XMFLOAT2        vTexcoord;

    static const unsigned int       iNumElements = { 2 };

    static constexpr D3D11_INPUT_ELEMENT_DESC   Elements[] = {
        {"POSITION", 0, DXGI_FORMAT_R32G32B32_FLOAT, 0, 0, D3D11_INPUT_PER_VERTEX_DATA, 0},
        {"TEXCOORD", 0, DXGI_FORMAT_R32G32_FLOAT, 0, 12, D3D11_INPUT_PER_VERTEX_DATA, 0}
    };
}VTXTEX;

typedef struct tagVertexPositionNormalTexcoord
{
    XMFLOAT3        vPosition;
    XMFLOAT3        vNormal;
    XMFLOAT2        vTexcoord;

    static const unsigned int       iNumElements = { 3 };

    static constexpr D3D11_INPUT_ELEMENT_DESC   Elements[] = {
        {"POSITION", 0, DXGI_FORMAT_R32G32B32_FLOAT, 0, 0, D3D11_INPUT_PER_VERTEX_DATA, 0},
        {"NORMAL", 0, DXGI_FORMAT_R32G32B32_FLOAT, 0, 12, D3D11_INPUT_PER_VERTEX_DATA, 0},
        {"TEXCOORD", 0, DXGI_FORMAT_R32G32_FLOAT, 0, 24, D3D11_INPUT_PER_VERTEX_DATA, 0}
    };
}VTXNORTEX;
```

### VTXNORTEX 바이트 오프셋 설명

| 시맨틱 | 포맷 | 바이트 크기 | 오프셋 |
|--------|------|-------------|--------|
| POSITION | R32G32B32_FLOAT | 12 | 0 |
| NORMAL | R32G32B32_FLOAT | 12 | 12 |
| TEXCOORD | R32G32_FLOAT | 8 | 24 |

총 정점 크기: **32바이트** (`sizeof(VTXNORTEX)`)

## 3. CVIBuffer_Terrain 전면 재작성

### 이론/개념

**HeightMap**: 회색조 BMP 이미지에서 각 픽셀의 밝기(R 채널)를 높이 값으로 사용하여 3D 지형을 생성하는 기법입니다.

- BMP 파일의 `BITMAPINFOHEADER`에서 `biWidth`(가로 정점 수), `biHeight`(세로 정점 수)를 읽음
- 각 픽셀의 RGBA 값에서 `& 0x000000ff`로 R 채널만 추출하여 높이로 사용
- 높이값을 `/10.f`로 나눠 스케일 조절

**격자 인덱싱 패턴:**
```
(i+1)*X+j ─── (i+1)*X+j+1
    │ ╲           │
    │   ╲         │
    │     ╲       │
    │       ╲     │
    │         ╲   │
    │           ╲ │
  i*X+j ────── i*X+j+1
```
각 셀마다 삼각형 2개 = 인덱스 6개.
총 인덱스 수 = `(Width-1) * (Height-1) * 2 * 3`

### 코드 변경 (VIBuffer_Terrain.h)

**Before (A 프로젝트):**
```cpp
class ENGINE_DLL CVIBuffer_Terrain final : public CVIBuffer
{
    // m_iNumVerticesX, m_iNumVerticesZ 없음
    // 하드코딩 4정점 사각형
};
```

**After (B 수업):**
```cpp
class ENGINE_DLL CVIBuffer_Terrain final : public CVIBuffer
{
private:

private:

public:

    virtual void Free() override;

```

### 코드 변경 (VIBuffer_Terrain.cpp - Initialize_Prototype)

```cpp
HRESULT CVIBuffer_Terrain::Initialize_Prototype(const _tchar* pHeightMapFilePath)
{
    // 1. BMP 파일 읽기
    _ulong          dwByte = { };
    HANDLE          hFile = CreateFile(pHeightMapFilePath, GENERIC_READ, 0,
                        nullptr, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, nullptr);
    if (0 == hFile)
        return E_FAIL;

    BITMAPFILEHEADER    fh{};
    BITMAPINFOHEADER    ih{};
    _uint*              pPixels = { nullptr };

    ReadFile(hFile, &fh, sizeof fh, &dwByte, nullptr);
    ReadFile(hFile, &ih, sizeof ih, &dwByte, nullptr);

    m_iNumVerticesX = ih.biWidth;
    m_iNumVerticesZ = ih.biHeight;
    m_iNumVertices  = m_iNumVerticesX * m_iNumVerticesZ;

    pPixels = new _uint[m_iNumVertices];
    ZeroMemory(pPixels, sizeof(_uint) * m_iNumVertices);
    ReadFile(hFile, pPixels, sizeof(_uint) * m_iNumVertices, &dwByte, nullptr);
    CloseHandle(hFile);

    // 2. 버퍼 파라미터 설정
    m_iNumVertexBuffers = 1;
    m_iVertexStride     = sizeof(VTXNORTEX);
    m_iNumIndices       = (m_iNumVerticesX - 1) * (m_iNumVerticesZ - 1) * 2 * 3;
    m_iIndexStride      = 4;
    m_eIndexFormat      = DXGI_FORMAT_R32_UINT;
    m_ePrimitiveType    = D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST;

    // 3. 정점 생성 (HeightMap 기반)
    VTXNORTEX* pVertices = new VTXNORTEX[m_iNumVertices];
    ZeroMemory(pVertices, sizeof(VTXNORTEX) * m_iNumVertices);

    for (size_t i = 0; i < m_iNumVerticesZ; i++)
    {
        for (size_t j = 0; j < m_iNumVerticesX; j++)
        {
            size_t iIndex = i * m_iNumVerticesX + j;

            pVertices[iIndex].vPosition = _float3(
                j,
                (pPixels[iIndex] & 0x000000ff) / 10.f,   // R채널 → 높이
                i
            );
            pVertices[iIndex].vNormal   = _float3(0.f, 0.f, 0.f);  // 추후 계산
            pVertices[iIndex].vTexcoord = _float2(
                j / (m_iNumVerticesX - 1.f),
                i / (m_iNumVerticesZ - 1.f)
            );
        }
    }

    // 4. 인덱스 생성 (격자 삼각형)
    _uint* pIndices = new _uint[m_iNumIndices];
    ZeroMemory(pIndices, sizeof(_uint) * m_iNumIndices);

    _uint iNumIndices = {};
    for (size_t i = 0; i < m_iNumVerticesZ - 1; i++)
    {
        for (size_t j = 0; j < m_iNumVerticesX - 1; j++)
        {
            size_t iIndex = i * m_iNumVerticesX + j;

            _uint iIndices[4] = {
                iIndex + m_iNumVerticesX,       // 좌상
                iIndex + m_iNumVerticesX + 1,   // 우상
                iIndex + 1,                     // 우하
                iIndex                          // 좌하
            };

            // 삼각형 1 (좌상 → 우상 → 우하)
            pIndices[iNumIndices++] = iIndices[0];
            pIndices[iNumIndices++] = iIndices[1];
            pIndices[iNumIndices++] = iIndices[2];

            // 삼각형 2 (좌상 → 우하 → 좌하)
            pIndices[iNumIndices++] = iIndices[0];
            pIndices[iNumIndices++] = iIndices[2];
            pIndices[iNumIndices++] = iIndices[3];
        }
    }

    // 5. D3D11 버퍼 생성 (VB)
    D3D11_BUFFER_DESC       VBDesc{};
    VBDesc.ByteWidth    = m_iVertexStride * m_iNumVertices;
    VBDesc.Usage        = D3D11_USAGE_DEFAULT;
    VBDesc.BindFlags    = D3D11_BIND_VERTEX_BUFFER;

    D3D11_SUBRESOURCE_DATA  VBSubResourceData{};
    VBSubResourceData.pSysMem = pVertices;

    m_pDevice->CreateBuffer(&VBDesc, &VBSubResourceData, &m_pVB);

    // 6. D3D11 버퍼 생성 (IB)
    D3D11_BUFFER_DESC       IBDesc{};
    IBDesc.ByteWidth    = m_iIndexStride * m_iNumIndices;
    IBDesc.Usage        = D3D11_USAGE_DEFAULT;
    IBDesc.BindFlags    = D3D11_BIND_INDEX_BUFFER;

    D3D11_SUBRESOURCE_DATA  IBSubResourceData{};
    IBSubResourceData.pSysMem = pIndices;

    m_pDevice->CreateBuffer(&IBDesc, &IBSubResourceData, &m_pIB);

    // 7. CPU측 임시 데이터 해제
    Safe_Delete_Array(pVertices);
    Safe_Delete_Array(pIndices);
    Safe_Delete_Array(pPixels);

    return S_OK;
}
```

### 핵심 포인트

- **VTXNORTEX** 사용 (Normal 포함) — 향후 라이팅 계산의 기반
- **32비트 인덱스** (`DXGI_FORMAT_R32_UINT`) — 65535개 초과 정점 지원
- Normal은 `(0,0,0)`으로 초기화 — 향후 수업에서 노멀 계산 로직 추가 예정
- `& 0x000000ff`로 R채널 추출 후 `/10.f`로 높이 스케일링

## 4. Shader_VtxNorTex.hlsl 신규 추가

### 이론/개념

`VTXNORTEX` 정점 포맷에 대응하는 새 셰이더입니다. 기존 `Shader_VtxTex.hlsl`과의 차이는 `VS_IN`에 `NORMAL` 시맨틱이 추가된 것입니다.
현재는 Normal을 사용하지 않지만, 향후 라이팅(디퓨즈/스펙큘러) 계산의 기반이 됩니다.

### 전체 코드 (Client/Bin/ShaderFiles/Shader_VtxNorTex.hlsl)

```hlsl
float4x4 g_WorldMatrix, g_ViewMatrix, g_ProjMatrix;
texture2D g_Texture;

sampler DefaultSampler = sampler_state
{
    filter = min_mag_mip_linear;
    AddressU = wrap;
    AddressV = wrap;
};

struct VS_IN
{
    float3 vPosition : POSITION;
    float3 vNormal : NORMAL;           // VTXNORTEX 대응 - 추가된 부분
    float2 vTexcoord : TEXCOORD0;
};

struct VS_OUT
{
    float4 vPosition : SV_POSITION;
    float2 vTexcoord : TEXCOORD0;
};

VS_OUT VS_MAIN(VS_IN In)
{
    VS_OUT Out;

    float4 vPosition = mul(float4(In.vPosition, 1.f), g_WorldMatrix);
    vPosition = mul(vPosition, g_ViewMatrix);
    vPosition = mul(vPosition, g_ProjMatrix);

    Out.vPosition = vPosition;
    Out.vTexcoord = In.vTexcoord;

    return Out;
}

struct PS_IN
{
    float4 vPosition : SV_POSITION;
    float2 vTexcoord : TEXCOORD0;
};

struct PS_OUT
{
    float4 vColor : SV_TARGET0;
};

PS_OUT PS_MAIN(PS_IN In)
{
    PS_OUT Out;

    Out.vColor = g_Texture.Sample(DefaultSampler, In.vTexcoord);
    if (Out.vColor.a < 0.1f)
        discard;

    return Out;
}

technique11 DefaultTechnique
{
    pass DefaultPass
    {
        VertexShader = compile vs_5_0 VS_MAIN();
        PixelShader = compile ps_5_0 PS_MAIN();
    }
}
```

### Shader_VtxTex.hlsl과의 차이 비교

| 항목 | Shader_VtxTex | Shader_VtxNorTex |
|------|---------------|-------------------|
| VS_IN | Position + Texcoord | Position + **Normal** + Texcoord |
| Normal 사용 여부 | 없음 | 입력만 받고 미사용 (향후 라이팅용) |
| PS_MAIN | `vColor.gb = vColor.r;` 포함 | 순수 텍스처 색상 출력 |

## 5. CCamera 엔진 클래스 신규 추가

### 이론/개념

카메라는 게임 세계를 바라보는 시점을 결정합니다. 엔진 레벨에서 `abstract` 클래스로 뼈대만 제공하고, 구체적인 카메라(자유시점, 3인칭 등)는 Client에서 구현합니다.

**CAMERA_DESC 상속 체인:**
```
TRANSFORM_DESC → GAMEOBJECT_DESC → CAMERA_DESC
                                    + vEye (카메라 위치)
                                    + vAt (바라보는 지점)
                                    + fFovy (시야각)
                                    + fNear (근평면)
                                    + fFar (원평면)
```

**Initialize 흐름:**
1. DESC에서 Near/Far/Fovy 파싱
2. `__super::Initialize(pArg)` → CTransform 생성
3. `Set_State(POSITION, vEye)` → 카메라 위치 설정
4. `LookAt(vAt)` → 카메라 방향 설정

### 코드 (Engine/Public/Camera.h)

```cpp
#pragma once

#include "GameObject.h"

NS_BEGIN(Engine)

class ENGINE_DLL CCamera abstract : public CGameObject
{
public:
    typedef struct tagCameraDesc : public CGameObject::GAMEOBJECT_DESC
    {
        _float3         vEye, vAt;
        _float          fFovy, fNear, fFar;
    }CAMERA_DESC;

protected:
    CCamera(ID3D11Device* pDevice, ID3D11DeviceContext* pContext);
    CCamera(const CCamera& Prototype);
    virtual ~CCamera() = default;

public:
    virtual HRESULT Initialize_Prototype();
    virtual HRESULT Initialize(void* pArg);
    virtual void Priority_Update(_float fTimeDelta);
    virtual void Update(_float fTimeDelta);
    virtual void Late_Update(_float fTimeDelta);
    virtual HRESULT Render();

private:
    _float          m_fFovy, m_fNear, m_fFar;
    class CPipeLine* m_pPipeLine = { nullptr };

public:
    virtual CGameObject* Clone(void* pArg) = 0;
    virtual void Free() override;
};

NS_END
```

### 코드 (Engine/Private/Camera.cpp)

```cpp
#include "Camera.h"

CCamera::CCamera(ID3D11Device* pDevice, ID3D11DeviceContext* pContext)
    : CGameObject{ pDevice, pContext }
{
}

    : CGameObject{ Prototype }

    m_fNear  = pDesc->fNear;
    m_fFar   = pDesc->fFar;
    m_fFovy  = pDesc->fFovy;

    if (FAILED(__super::Initialize(pArg)))
        return E_FAIL;

    // 카메라 위치 설정

        XMVectorSetW(XMLoadFloat3(&pDesc->vEye), 1.f));

    // 카메라가 바라보는 방향 설정
    m_pTransformCom->LookAt(
        XMVectorSetW(XMLoadFloat3(&pDesc->vAt), 1.f));

    return S_OK;
}

void CCamera::Priority_Update(_float fTimeDelta)
{
}

void CCamera::Update(_float fTimeDelta)
{
}

void CCamera::Late_Update(_float fTimeDelta)
{
}

{
    return E_NOTIMPL;
}

CGameObject* CCamera::Clone(void* pArg)
{
    return nullptr;
}

{
}
```

### 현재 상태 및 향후 과제

- `CPipeLine*` 멤버가 선언만 되어 있고 사용하지 않음 → 다음 수업에서 View/Proj 행렬을 엔진 전역으로 관리하는 `CPipeLine` 클래스 도입 예정
- `Clone()`이 `nullptr` 반환 → Client에서 구체 카메라 클래스를 만들어야 함
- `Free()`가 비어 있음 → `m_pPipeLine` 해제 등 미구현
- `Render()`가 `E_NOTIMPL` 반환 → 카메라는 렌더링 대상이 아니므로 정상

## 6. Loader - GamePlay 리소스 등록 변경

### 이론/개념

Terrain이 `VTXNORTEX` 정점 + HeightMap 기반으로 변경되었으므로, Loader에서 해당 프로토타입들을 등록해야 합니다.

### 코드 변경 (Loader.cpp - Ready_Resources_For_GamePlay)

**Before (A 프로젝트):**
```cpp
HRESULT CLoader::Ready_Resources_For_GamePlay()
{
```

## 9. 수업 메모 - 인덱스 크기 최적화 고려사항

수업 폴더에 두 개의 .txt 파일이 포함되어 있습니다. 인덱스 버퍼의 바이트 크기를 최적화하는 과정에서의 고민과 실험이 담긴 코드입니다.

### 9.1 인덱스 버퍼란?

정점 버퍼(Vertex Buffer)는 메시의 각 정점 데이터를 저장하고, 인덱스 버퍼(Index Buffer)는 **어떤 정점들을 어떤 순서로 조합하여 삼각형을 만들지**를 지정합니다.

```
정점 버퍼:   [V0] [V1] [V2] [V3]
인덱스 버퍼: [0, 1, 2, 0, 2, 3]  <- 삼각형 2개 (V0-V1-V2, V0-V2-V3)
```

인덱스 하나는 **정점 배열의 인덱스(번호)** 이므로, 정점이 몇 개인지에 따라 필요한 숫자 범위가 달라집니다.

### 9.2 16비트 vs 32비트 인덱스

| 항목 | 16비트 (`_ushort`) | 32비트 (`_uint`) |
|------|---------------------|-------------------|
| DXGI 포맷 | `DXGI_FORMAT_R16_UINT` | `DXGI_FORMAT_R32_UINT` |
| 바이트 크기 | 2 bytes | 4 bytes |
| 표현 범위 | 0 ~ 65,535 | 0 ~ 4,294,967,295 |
| 메모리 사용 | 기준 | 2배 |

**핵심**: 정점 수가 65,535개 이하이면 16비트로 충분하지만, 초과하면 반드시 32비트를 사용해야 합니다.

**HeightMap 지형의 경우:**
- 256x256 HeightMap -> 65,536 정점 -> 16비트 **초과** (32비트 필요)
- 128x128 HeightMap -> 16,384 정점 -> 16비트 **충분**

즉, HeightMap 해상도에 따라 인덱스 크기를 분기해야 할 수도 있습니다. 이 고민이 두 .txt 파일의 핵심 주제입니다.

### 9.3 접근법 A: 무조건 4바이트 (VIBuffer_Terrain_4Byte.txt - 최종 채택)

```cpp
// 고정 4바이트
m_iIndexStride = 4;
m_eIndexFormat = DXGI_FORMAT_R32_UINT;

// _uint 배열로 직접 생성
_uint* pIndices = new _uint[m_iNumIndices];
ZeroMemory(pIndices, sizeof(_uint) * m_iNumIndices);

_uint iNumIndices = {};

for (size_t i = 0; i < m_iNumVerticesZ - 1; i++)
{
    for (size_t j = 0; j < m_iNumVerticesX - 1; j++)
    {
        size_t iIndex = i * m_iNumVerticesX + j;

        _uint iIndices[4] = {
            iIndex + m_iNumVerticesX,       // 좌상
            iIndex + m_iNumVerticesX + 1,   // 우상
            iIndex + 1,                     // 우하
            iIndex                          // 좌하
        };

        // 삼각형 1
        pIndices[iNumIndices++] = iIndices[0];
        pIndices[iNumIndices++] = iIndices[1];
        pIndices[iNumIndices++] = iIndices[2];

        // 삼각형 2
        pIndices[iNumIndices++] = iIndices[0];
        pIndices[iNumIndices++] = iIndices[2];
        pIndices[iNumIndices++] = iIndices[3];
    }
}
```

**특징:**
- 타입이 `_uint*`로 명확 -> 직접 인덱싱(`pIndices[i] = value`) 가능
- 정점 수에 관계없이 항상 동작
- 코드가 단순하고 안전

### 9.4 접근법 B: 동적 분기 (VIBuffer_Terrains.txt - 실험적, 미채택)

#### 9.4.1 핵심 아이디어

정점 수에 따라 인덱스 크기를 런타임에 결정합니다:

```cpp
m_iIndexStride = m_iNumVertices > 65000 ? 4 : 2;
m_eIndexFormat = m_iIndexStride == 2 ? DXGI_FORMAT_R16_UINT : DXGI_FORMAT_R32_UINT;
```

> **65000 vs 65535**: 실제 한계값은 65,535(`_ushort` 최대값)이지만, 수업에서 65000으로 여유를 둔 것은 안전 마진입니다.

#### 9.4.2 void* 를 사용하는 이유

인덱스 크기가 2바이트일 수도, 4바이트일 수도 있으므로 특정 타입의 포인터(`_ushort*` 또는 `_uint*`)로 고정할 수 없습니다. 이때 **타입을 지정하지 않는 `void*`** 를 사용합니다:

```cpp
void* pIndices = new _ushort[m_iNumIndices];   // 일단 최소 크기로 할당
```

`void*`는 **어떤 타입의 데이터든 가리킬 수 있는 범용 포인터**입니다. 하지만 `void*`를 직접 인덱싱(`pIndices[i]`)할 수는 없습니다. C/C++에서 배열 인덱싱은 포인터 산술(`ptr + i * sizeof(타입)`)인데, `void`는 크기가 정의되지 않은 타입이기 때문입니다.

#### 9.4.3 바이트 단위 접근 (memcpy 방식)

`void*`를 `_byte*` (`unsigned char*`)로 캐스팅하면 **1바이트 단위**로 메모리에 접근할 수 있습니다:

```cpp
auto pIndex = static_cast<_byte*>(pIndices);

// m_iIndexStride 바이트만큼 복사
memcpy(pIndex + m_iIndexStride * iNumIndices++, &iIndices[0], m_iIndexStride);
```

**동작 원리:**
```
m_iIndexStride = 2 (_ushort)인 경우:
  메모리: [AA BB] [CC DD] [EE FF] ...
  pIndex + 2 * 0 -> [AA BB] (인덱스 0)
  pIndex + 2 * 1 -> [CC DD] (인덱스 1)

m_iIndexStride = 4 (_uint)인 경우:
  메모리: [AA BB CC DD] [EE FF GG HH] ...
  pIndex + 4 * 0 -> [AA BB CC DD] (인덱스 0)
  pIndex + 4 * 1 -> [EE FF GG HH] (인덱스 1)
```

`memcpy(&dest, &src, m_iIndexStride)`는 **m_iIndexStride 바이트만큼만 복사**하므로:
- 2바이트 모드: `_uint` 값에서 하위 2바이트만 복사 (값이 65535 이하이므로 상위 2바이트는 0)
- 4바이트 모드: `_uint` 값 전체 복사

이렇게 하면 하나의 코드로 2바이트/4바이트 인덱스 모두 처리할 수 있습니다.

#### 9.4.4 실험 코드의 문제점

수업 .txt 파일의 코드는 **미완성 실험 코드**로, 여러 접근법이 섞여 있습니다:

```cpp
// 접근법 1: memcpy (올바른 방향)
auto pIndex = static_cast<_byte*>(pIndices);
memcpy(pIndex + m_iIndexStride * iNumIndices++, &iIndices[0], m_iIndexStride);

// 접근법 2: _byte 직접 대입 (잘못됨 - 1바이트만 기록됨)
pIndex[m_iIndexStride * iNumIndices++] = iIndices[0];  // 하위 1바이트만 저장!
pIndex[m_iIndexStride * iNumIndices++] = 1;            // 리터럴 대입
pIndex[m_iIndexStride * iNumIndices++] = 2;

// 접근법 3: void* 직접 인덱싱 (컴파일 에러)
pIndices[0] = 0;   // void*는 인덱싱 불가!
pIndices[1] = 1;
```

| 접근법 | 문제 |
|--------|------|
| memcpy | 정상 동작. `m_iIndexStride` 바이트만큼 정확히 복사 |
| `_byte` 직접 대입 | `_byte`는 1바이트이므로, 2바이트/4바이트 인덱스의 **하위 1바이트만 저장**됨. 256 이상의 인덱스 값이 잘림 |
| `void*` 인덱싱 | `void*`는 크기 정보가 없어 컴파일 에러. `void*`로 산술 연산 불가 |

#### 9.4.5 올바른 동적 분기 구현 (참고)

만약 동적 분기를 올바르게 구현한다면 두 가지 방법이 있습니다:

**방법 1: memcpy 일관 사용**
```cpp
void* pIndices = malloc(m_iIndexStride * m_iNumIndices);
auto pRaw = static_cast<_byte*>(pIndices);

_uint iNumIdx = 0;
for (/* 격자 순회 */)
{
    _uint iIndices[4] = { /* 좌상, 우상, 우하, 좌하 */ };

    memcpy(pRaw + m_iIndexStride * iNumIdx++, &iIndices[0], m_iIndexStride);
    memcpy(pRaw + m_iIndexStride * iNumIdx++, &iIndices[1], m_iIndexStride);
    memcpy(pRaw + m_iIndexStride * iNumIdx++, &iIndices[2], m_iIndexStride);

    memcpy(pRaw + m_iIndexStride * iNumIdx++, &iIndices[0], m_iIndexStride);
    memcpy(pRaw + m_iIndexStride * iNumIdx++, &iIndices[2], m_iIndexStride);
    memcpy(pRaw + m_iIndexStride * iNumIdx++, &iIndices[3], m_iIndexStride);
}
```

**방법 2: if 분기로 타입별 처리**
```cpp
if (m_iIndexStride == 2)
{
    _ushort* pIndices = new _ushort[m_iNumIndices];
    // _ushort 기반 직접 대입...
}
else
{
    _uint* pIndices = new _uint[m_iNumIndices];
    // _uint 기반 직접 대입...
}
```

방법 2는 코드 중복이 생기지만 가독성이 좋고, 방법 1은 중복 없이 유연하지만 memcpy 호출 오버헤드가 있습니다.

### 9.5 메모리 절약 효과 비교

구체적인 수치로 비교하면:

```
128x128 HeightMap (16,384 정점):
  인덱스 수 = 127 x 127 x 2 x 3 = 96,774개
  16비트: 96,774 x 2 = 193,548 bytes = 약 189 KB
  32비트: 96,774 x 4 = 387,096 bytes = 약 378 KB
  -> 절약량: 약 189 KB

256x256 HeightMap (65,536 정점):
  인덱스 수 = 255 x 255 x 2 x 3 = 390,150개
  32비트: 390,150 x 4 = 1,560,600 bytes = 약 1.49 MB
  (16비트 사용 불가 - 정점 65,536개로 한계 초과)
```

### 9.6 실무적 판단

| 고려 요소 | 2바이트 | 4바이트 |
|-----------|---------|---------|
| GPU 메모리 | 절약 | 여유로움 |
| GPU 대역폭 | 약간 유리 | 약간 불리 |
| 코드 복잡도 | 높음 (분기/memcpy) | 낮음 (직접 대입) |
| 유지보수 | 버그 유발 가능 | 안전 |
| 확장성 | 메시 커지면 4바이트로 교체 필요 | 제한 없음 |

**수업 결론:** 현대 GPU에서 인덱스 버퍼의 메모리 차이는 미미하고, HeightMap 해상도가 256x256만 되어도 16비트가 불가능합니다. 코드 안정성과 단순함을 위해 **4바이트 방식을 채택**했습니다.

### 9.7 관련 개념 정리

#### `& 0x000000ff` 비트 마스킹

BMP 파일의 각 픽셀은 32비트(BGRA)로 저장됩니다:

```
픽셀 데이터:    AAAA AAAA  RRRR RRRR  GGGG GGGG  BBBB BBBB
               (Alpha)    (Red)      (Green)    (Blue)
```

메모리에서 `_uint`로 읽으면 리틀 엔디안 기준으로:

```
_uint 값:      AARRGGBB (hex)

& 0x000000FF:  0x000000BB -> Blue 채널만 추출
```

회색조 이미지에서는 R=G=B이므로 어떤 채널을 추출해도 동일한 높이값을 얻습니다.

#### 엔디안(Endianness)과 memcpy

`memcpy`로 `_uint`에서 2바이트만 복사할 때, x86(리틀 엔디안)에서는 **하위 바이트가 먼저** 저장됩니다:

```
_uint value = 0x00001234;

메모리 레이아웃 (리틀 엔디안):
주소: [낮음] -> [높음]
내용: [34] [12] [00] [00]

memcpy(dest, &value, 2):
복사: [34] [12] -> 0x1234 (올바른 값)
```

따라서 리틀 엔디안 시스템에서는 `memcpy`로 하위 N바이트를 안전하게 추출할 수 있습니다. 단, 이 방식은 **플랫폼 의존적**이며, 빅 엔디안 시스템에서는 상위 바이트가 먼저 복사되어 잘못된 값이 됩니다. (x86/x64 Windows 환경에서는 항상 리틀 엔디안이므로 문제없음)
