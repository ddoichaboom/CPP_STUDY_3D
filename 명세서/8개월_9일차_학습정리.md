# 8개월차 9일차 - Light System, Model Loading (Assimp)

## 개요

8일차에서 HLSL 셰이더에 하드코딩된 라이팅 변수를 사용했다면, 9일차에서는 **엔진 레벨의 조명 관리 시스템**을 구축하고, **3D 모델 로딩의 첫 발판**(Assimp)을 세운다.

- **Light System**: `LIGHT_DESC` 구조체 + `CLight` + `CLight_Manager` 로 조명을 엔진 서브시스템으로 승격
- **CGameInstance 위임**: `Add_Light()`, `Get_LightDesc()` 퍼사드 메서드 추가
- **Terrain 바인딩 변경**: 하드코딩 → `CLight_Manager`에서 조명 데이터를 가져와 셰이더에 바인딩
- **CModel 컴포넌트**: Assimp 라이브러리를 통한 FBX 모델 로딩 기초 골격
- **리소스 확장**: FBX 모델 파일(Fiona, ForkLift, Rock 등), DirectXTK 전체 헤더, Assimp 라이브러리 추가

## 1. 이론: 엔진 레벨 조명 관리의 필요성

### 1.1 8일차의 한계 — 하드코딩된 조명

8일차에서는 HLSL 셰이더 안에 조명 파라미터를 **기본값으로 하드코딩**했다:

```hlsl
// 8일차 — 셰이더에 값이 박혀 있음
vector g_vLightDir      = vector(1.f, -1.f, 1.f, 0.f);
vector g_vLightDiffuse  = vector(1.f, 1.f, 1.f, 1.f);
vector g_vLightAmbient  = vector(1.f, 1.f, 1.f, 1.f);
vector g_vLightSpecular = vector(1.f, 1.f, 1.f, 1.f);
```

이 방식의 문제:
- **변경 불가**: 런타임에 빛 방향이나 색상을 바꿀 수 없다
- **다중 광원 불가**: 광원이 하나로 고정됨
- **레벨별 차이 불가**: 모든 레벨에서 동일한 조명
- **C++ 코드와 분리**: 엔진이 조명을 관리하지 않으므로 게임 로직과 연동 불가

### 1.2 9일차의 해결 — 엔진 서브시스템으로 승격

```
하드코딩 (셰이더 내부)
    ↓
LIGHT_DESC (C++ 구조체로 데이터 정의)
    ↓
CLight (개별 광원 인스턴스)
    ↓
CLight_Manager (광원 목록 관리, 인덱스로 접근)
    ↓
CGameInstance (퍼사드 패턴으로 외부 노출)
    ↓
Client에서 Level 초기화 시 Add_Light() → Terrain::Render()에서 Get_LightDesc()로 셰이더에 바인딩
```

이 구조의 장점:
- 런타임에 조명 속성 변경 가능
- 여러 광원 추가 가능 (리스트 기반)
- 레벨마다 다른 조명 설정 가능
- 기존 엔진 패턴(싱글톤, 퍼사드, CBase 참조 카운팅)과 일관성 유지

## 2. LIGHT_DESC 구조체 & LIGHT 열거형

### 2.1 LIGHT 열거형 (Engine_Enum.h에 추가)

```cpp
// === Engine_Enum.h ===
enum class LIGHT { DIRECTIONAL, POINT, END };
```

| 타입 | 의미 | 특성 |
|------|------|------|
| `DIRECTIONAL` | 방향광 (태양) | 위치 없음, 방향만 있음, 모든 곳에 균일하게 비춤 |
| `POINT` | 점광원 (전구) | 위치 있음, 사방으로 방사, 거리에 따라 감쇠 |

### 2.2 LIGHT_DESC 구조체 (Engine_Struct.h에 추가)

```cpp
// === Engine_Struct.h ===
typedef struct tagLightDesc
{
    LIGHT       eType;                              // 광원 타입 (DIRECTIONAL / POINT)
    XMFLOAT4    vDiffuse, vAmbient, vSpecular;      // 빛의 3가지 색상 성분
    XMFLOAT4    vDirection;                         // 방향광용: 빛이 향하는 방향
    XMFLOAT4    vPosition;                          // 점광원용: 광원 위치
    float       fRange;                             // 점광원용: 유효 범위
}LIGHT_DESC;
```

**핵심 포인트:**
- `XMFLOAT4`를 사용하는 이유: HLSL의 `vector`(float4)와 1:1 대응. `Bind_RawValue`로 바이트 단위 복사 시 크기가 일치해야 한다
- `vDirection`과 `vPosition`은 모두 존재하지만, `eType`에 따라 하나만 의미를 가진다
  - `DIRECTIONAL`: `vDirection` 사용 (예: (1, -1, 1, 0) → 오른쪽 아래 뒤쪽으로 향하는 빛)
  - `POINT`: `vPosition` + `fRange` 사용
- 기존 `ENGINE_DESC`와 `VTXTEX`/`VTXNORTEX` 사이에 배치

### 2.3 Engine_Struct.h 전체 변경


## 3. CLight 클래스 — 개별 광원

### 3.1 설계 원칙

- `CBase` 상속 → 참조 카운팅 적용
- 단순 데이터 홀더: `LIGHT_DESC`를 저장하고 외부에 `const LIGHT_DESC*`를 반환
- `CComponent`가 아닌 `CBase` 상속 → 프로토타입/클론 패턴 불필요 (광원은 복제가 아닌 생성)
- `ENGINE_DLL` 미사용 → 엔진 내부에서만 사용 (CGameInstance를 통해서만 접근)

### 3.2 Light.h

```cpp
#pragma once
#include "Base.h"

NS_BEGIN(Engine)

class CLight final : public CBase
{
private:
    CLight(ID3D11Device* pDevice, ID3D11DeviceContext* pContext);
    virtual ~CLight() = default;

public:
    const LIGHT_DESC* Get_LightDesc() const {
        return &m_LightDesc;
    }

public:
    HRESULT Initialize(const LIGHT_DESC& LightDesc);

private:
    ID3D11Device*           m_pDevice = { };
    ID3D11DeviceContext*    m_pContext = { };
    LIGHT_DESC              m_LightDesc{};

public:
    static CLight* Create(ID3D11Device* pDevice, ID3D11DeviceContext* pContext, const LIGHT_DESC& LightDesc);
    virtual void Free() override;
};

NS_END
```

### 3.3 Light.cpp

```cpp
#include "Light.h"

CLight::CLight(ID3D11Device* pDevice, ID3D11DeviceContext* pContext)
    : m_pDevice{ pDevice }
    , m_pContext{ pContext }
{
    Safe_AddRef(m_pDevice);
    Safe_AddRef(m_pContext);
}

HRESULT CLight::Initialize(const LIGHT_DESC& LightDesc)
{
    // LIGHT_DESC를 그대로 멤버에 복사
    m_LightDesc = LightDesc;
    return S_OK;
}

CLight* CLight::Create(ID3D11Device* pDevice, ID3D11DeviceContext* pContext, const LIGHT_DESC& LightDesc)
{
    CLight* pInstance = new CLight(pDevice, pContext);

    if (FAILED(pInstance->Initialize(LightDesc)))
    {
        MSG_BOX("Failed to Created : CLight");
        Safe_Release(pInstance);
    }

    return pInstance;
}

void CLight::Free()
{
    __super::Free();
    Safe_Release(m_pDevice);
    Safe_Release(m_pContext);
}
```

**핵심 분석:**
- 생성 패턴: `Create()` 정적 함수에서 `new` + `Initialize()` → 실패 시 `Safe_Release()`
- `Device`/`Context` 보유 이유: 현재는 사용하지 않지만, 향후 광원 관련 GPU 리소스(Shadow Map 등)를 위한 확장 여지
- `Free()`: Device/Context를 해제. `__super::Free()` 먼저 호출 (CBase 패턴)

## 4. CLight_Manager 클래스 — 광원 관리자

### 4.1 설계 원칙

- `CBase` 상속, `final` 키워드
- `list<CLight*>`로 광원 관리 — 동적 추가 가능, 인덱스로 접근
- 싱글톤이 아님 — `CGameInstance`가 하나의 인스턴스를 소유

### 4.2 Light_Manager.h

```cpp
#pragma once
#include "Base.h"

NS_BEGIN(Engine)

class CLight_Manager final : public CBase
{
private:
    CLight_Manager(ID3D11Device* pDevice, ID3D11DeviceContext* pContext);
    virtual ~CLight_Manager() = default;

public:
    const LIGHT_DESC* Get_LightDesc(_uint iIndex);

public:
    HRESULT Add_Light(const LIGHT_DESC& LightDesc);

private:
    ID3D11Device*           m_pDevice = { nullptr };
    ID3D11DeviceContext*    m_pContext = { nullptr };
    list<class CLight*>     m_Lights;

public:
    static CLight_Manager* Create(ID3D11Device* pDevice, ID3D11DeviceContext* pContext);
    virtual void Free() override;
};

NS_END
```

### 4.3 Light_Manager.cpp

```cpp
#include "Light_Manager.h"
#include "Light.h"

CLight_Manager::CLight_Manager(ID3D11Device* pDevice, ID3D11DeviceContext* pContext)
    : m_pDevice{ pDevice }
    , m_pContext{ pContext }
{
    Safe_AddRef(m_pDevice);
    Safe_AddRef(m_pContext);
}

const LIGHT_DESC* CLight_Manager::Get_LightDesc(_uint iIndex)
{
    // (1) 범위 검사
    if (iIndex >= m_Lights.size())
        return nullptr;

    // (2) list 순회로 iIndex번째 요소에 접근
    auto iter = m_Lights.begin();
    for (size_t i = 0; i < iIndex; i++)
        ++iter;

    if (iter == m_Lights.end())
        return nullptr;

    // (3) CLight의 LIGHT_DESC 포인터 반환
    return (*iter)->Get_LightDesc();
}

HRESULT CLight_Manager::Add_Light(const LIGHT_DESC& LightDesc)
{
    // CLight 인스턴스 생성 → 리스트에 추가
    CLight* pLight = CLight::Create(m_pDevice, m_pContext, LightDesc);
    if (nullptr == pLight)
        return E_FAIL;

    m_Lights.push_back(pLight);
    return S_OK;
}

CLight_Manager* CLight_Manager::Create(ID3D11Device* pDevice, ID3D11DeviceContext* pContext)
{
    return new CLight_Manager(pDevice, pContext);
}

void CLight_Manager::Free()
{
    __super::Free();

    // 모든 CLight 해제
    for (auto& pLight : m_Lights)
        Safe_Release(pLight);
    m_Lights.clear();

    Safe_Release(m_pDevice);
    Safe_Release(m_pContext);
}
```

### 4.4 왜 `list`인가?

| 자료구조 | 장점 | 단점 |
|----------|------|------|
| `vector` | 인덱스 접근 O(1) | 중간 삽입/삭제 비효율 |
| `list` | 중간 삽입/삭제 O(1) | 인덱스 접근 O(n) |

광원은 자주 추가/삭제될 수 있고, 인덱스 접근은 프레임당 소수 호출이므로 `list`가 적절. 광원 개수가 매우 많아지면 `vector`가 나을 수 있다.

### 4.5 `Get_LightDesc(iIndex)` 동작 흐름

```
Get_LightDesc(0)
  → m_Lights.begin() → CLight[0]->Get_LightDesc()
  → const LIGHT_DESC* 반환 (vDiffuse, vAmbient, vSpecular, vDirection 등)
```

### 6.3 Ready_Lights() 구현

```cpp
HRESULT CLevel_GamePlay::Ready_Lights()
{
    LIGHT_DESC LightDesc{};

    LightDesc.eType = LIGHT::DIRECTIONAL;
    LightDesc.vDiffuse  = _float4(1.f, 1.f, 1.f, 1.f);     // 흰색 확산광
    LightDesc.vAmbient  = _float4(1.f, 1.f, 1.f, 1.f);     // 흰색 환경광
    LightDesc.vSpecular = _float4(1.f, 1.f, 1.f, 1.f);     // 흰색 반사광
    LightDesc.vDirection = _float4(1.f, -1.f, 1.f, 0.f);   // 대각선 아래 방향

    if (FAILED(m_pGameInstance->Add_Light(LightDesc)))
        return E_FAIL;

    return S_OK;
}
```

**핵심 포인트:**
- `vDirection`의 `w = 0.f`: 방향 벡터이므로 이동 성분 없음
- `(1, -1, 1)`: 오른쪽+아래+앞쪽으로 향하는 빛. 셰이더에서 `-1`을 곱해 반대 방향으로 만들어 노멀과 내적
- 모든 색상이 `(1,1,1,1)` 흰색 → 텍스처 원래 색이 그대로 보이면서 조명 효과만 적용
- 향후 여러 광원 추가 가능: `Add_Light()`를 여러 번 호출하면 됨

```cpp
HRESULT CTerrain::Bind_ShaderResources()
{
    // === 기존과 동일 ===
    if (FAILED(m_pTransformCom->Bind_ShaderResource(m_pShaderCom, "g_WorldMatrix")))
        return E_FAIL;
    if (FAILED(m_pShaderCom->Bind_Matrix("g_ViewMatrix", m_pGameInstance->Get_Transform(D3DTS::VIEW))))
        return E_FAIL;
    if (FAILED(m_pShaderCom->Bind_Matrix("g_ProjMatrix", m_pGameInstance->Get_Transform(D3DTS::PROJ))))
        return E_FAIL;
    if (FAILED(m_pTextureCom->Bind_ShaderResource(m_pShaderCom, "g_DiffuseTexture", 0)))
        return E_FAIL;
    if (FAILED(m_pShaderCom->Bind_RawValue("g_vCamPosition", m_pGameInstance->Get_CamPosition(), sizeof(_float4))))
        return E_FAIL;

    // ★ NEW: CLight_Manager에서 조명 데이터를 가져와 바인딩
    const LIGHT_DESC* pLightDesc = m_pGameInstance->Get_LightDesc(0);
    if (nullptr == pLightDesc)
        return E_FAIL;

    if (FAILED(m_pShaderCom->Bind_RawValue("g_vLightDir", &pLightDesc->vDirection, sizeof(_float4))))
        return E_FAIL;
    if (FAILED(m_pShaderCom->Bind_RawValue("g_vLightDiffuse", &pLightDesc->vDiffuse, sizeof(_float4))))
        return E_FAIL;
    if (FAILED(m_pShaderCom->Bind_RawValue("g_vLightAmbient", &pLightDesc->vAmbient, sizeof(_float4))))
        return E_FAIL;
    if (FAILED(m_pShaderCom->Bind_RawValue("g_vLightSpecular", &pLightDesc->vSpecular, sizeof(_float4))))
        return E_FAIL;

    return S_OK;
}
```

### 7.3 데이터 흐름 전체 정리

```
Level_GamePlay::Ready_Lights()
    LIGHT_DESC 생성 (vDiffuse, vAmbient, vSpecular, vDirection)
    → m_pGameInstance->Add_Light(LightDesc)
        → CLight_Manager::Add_Light()
            → CLight::Create() — LIGHT_DESC 저장
            → m_Lights.push_back(pLight)

매 프레임 Terrain::Render() → Bind_ShaderResources()
    → m_pGameInstance->Get_LightDesc(0)
        → CLight_Manager::Get_LightDesc(0)
            → m_Lights.begin() → CLight::Get_LightDesc()
            → const LIGHT_DESC* 반환
    → Bind_RawValue("g_vLightDir", &pLightDesc->vDirection, 16)
    → Bind_RawValue("g_vLightDiffuse", &pLightDesc->vDiffuse, 16)
    → Bind_RawValue("g_vLightAmbient", &pLightDesc->vAmbient, 16)
    → Bind_RawValue("g_vLightSpecular", &pLightDesc->vSpecular, 16)
        → SetRawValue() → HLSL 전역 변수에 바이트 복사
```

## 8. Shader_VtxNorTex.hlsl — 변경 사항

### 8.1 핵심 변경: 기본값 제거

8일차에서는 셰이더 전역 변수에 기본값이 있었지만, 9일차에서는 C++ 코드가 매 프레임 바인딩하므로 **기본값이 불필요**해진다.

```hlsl
// === 9일차 Shader_VtxNorTex.hlsl ===

float4x4 g_WorldMatrix, g_ViewMatrix, g_ProjMatrix;
vector g_vCamPosition;

// 빛 — 기본값 없음! C++에서 Bind_RawValue로 매 프레임 전달
vector g_vLightDir;
vector g_vLightDiffuse;
vector g_vLightAmbient;
vector g_vLightSpecular;

// 재질 — 기본값 유지 (오브젝트마다 다를 수 있지만 현재는 전역)
texture2D g_DiffuseTexture;
vector g_vMtrlAmbient  = vector(0.4f, 0.4f, 0.4f, 1.f);
vector g_vMtrlSpecular = vector(1.f, 1.f, 1.f, 1.f);
```

**왜 재질(Material)은 기본값을 유지하는가?**
- 현재 단계에서는 오브젝트별 재질 시스템이 없음
- 모든 오브젝트가 동일한 재질 속성을 사용
- 향후 `MATERIAL_DESC` 구조체를 만들어 C++에서 바인딩하면 기본값도 제거

### 8.2 셰이더 전체 코드 (라이팅 부분은 8일차와 동일)

VS_MAIN, PS_MAIN의 계산 로직 자체는 8일차와 동일하다. 달라진 것은 **데이터의 출처**뿐:
- 8일차: HLSL 기본값 → 고정
- 9일차: C++ Bind_RawValue → 동적 변경 가능

## 9. 이론: Assimp 라이브러리와 3D 모델 로딩

### 9.1 Assimp이란?

**Assimp** (Open Asset Import Library)은 40개 이상의 3D 모델 포맷을 통일된 데이터 구조로 로딩하는 오픈소스 라이브러리.

```
지원 포맷:
  FBX, OBJ, GLTF, DAE(Collada), 3DS, PLY, STL ...
      ↓ Assimp::Importer::ReadFile()
  통일된 aiScene 구조
      ↓ 접근
  aiMesh, aiMaterial, aiAnimation, aiBone ...
```

### 9.2 핵심 Assimp 구조

```
aiScene (최상위 루트)
├── mMeshes[]          — 메시 배열 (정점/인덱스/UV/노멀)
├── mMaterials[]       — 재질 배열 (텍스처 경로, 색상 등)
├── mAnimations[]      — 애니메이션 배열
├── mTextures[]        — 임베디드 텍스처 (FBX 내장 텍스처)
└── mRootNode          — 노드 트리 (계층 구조, 변환 행렬)
    ├── mChildren[]    — 자식 노드들
    └── mMeshes[]      — 이 노드가 참조하는 메시 인덱스
```

### 9.3 좌표계 변환 문제

| 프로그램 | 좌표계 | 특성 |
|----------|--------|------|
| DirectX  | **왼손 좌표계** (Left-Handed) | Z축이 화면 안쪽으로 |
| OpenGL / Blender / Maya | **오른손 좌표계** (Right-Handed) | Z축이 화면 바깥으로 |
| FBX (대부분) | 오른손 좌표계 | 모델링 소프트웨어 기준 |

Assimp의 `aiProcess_ConvertToLeftHanded` 플래그가 자동으로 변환해준다:
- 삼각형 와인딩(winding) 순서 반전
- UV 좌표 Y축 반전
- 법선 벡터 방향 보정

### 9.4 후처리(Post-Processing) 플래그

```cpp
_uint iFlag = aiProcess_ConvertToLeftHanded | aiProcessPreset_TargetRealtime_Fast;
```

| 플래그 | 설명 |
|--------|------|
| `aiProcess_ConvertToLeftHanded` | DirectX 왼손 좌표계로 변환 |
| `aiProcessPreset_TargetRealtime_Fast` | 실시간 렌더링에 적합한 전처리 묶음 |

`aiProcessPreset_TargetRealtime_Fast`에 포함되는 주요 처리:
- `aiProcess_Triangulate`: 모든 폴리곤을 삼각형으로 분해
- `aiProcess_SortByPType`: 프리미티브 타입별 정렬
- `aiProcess_GenNormals`: 노멀이 없으면 자동 생성
- `aiProcess_GenUVCoords`: UV가 없으면 자동 생성

## 10. CModel 컴포넌트 — 기초 골격

### 10.1 설계 원칙

- `CComponent` 상속 → 프로토타입/클론 패턴 적용
- `ENGINE_DLL` 사용 → Client에서 직접 사용 가능
- 현재는 **골격(Skeleton)만 구현** — aiScene 로딩까지만. 메시/텍스처 추출은 향후

### 10.2 Model.h

```cpp
#pragma once
#include "Component.h"

NS_BEGIN(Engine)

class ENGINE_DLL CModel final : public CComponent
{
private:
    CModel(ID3D11Device* pDevice, ID3D11DeviceContext* pContext);
    CModel(const CModel& Prototype);
    virtual ~CModel() = default;

public:
    virtual HRESULT Initialize_Prototype(const _char* pModelFilePath);
    virtual HRESULT Initialize(void* pArg);

private:
    const aiScene*  m_pAIScene = { nullptr };    // Assimp이 파싱한 씬 데이터
    Importer        m_Importer = {};              // Assimp 임포터 (aiScene 수명 관리)

public:
    static CModel* Create(ID3D11Device* pDevice, ID3D11DeviceContext* pContext, const _char* pModelFilePath);
    virtual CComponent* Clone(void* pArg) override;
    virtual void Free() override;

NS_END
```

### 10.3 Model.cpp

```cpp
#include "Model.h"

CModel::CModel(ID3D11Device* pDevice, ID3D11DeviceContext* pContext)
    : CComponent{ pDevice, pContext }
{
}

CModel::CModel(const CModel& Prototype)
    : CComponent{ Prototype }
{
    // 주의: m_pAIScene과 m_Importer는 복사하지 않음
    // 복사 생성자에서 별도 처리 필요 (향후 구현)
}

HRESULT CModel::Initialize_Prototype(const _char* pModelFilePath)
{
    // (1) 후처리 플래그 설정
    _uint iFlag = { aiProcess_ConvertToLeftHanded |
                    aiProcessPreset_TargetRealtime_Fast };

    // (2) 모델 파일 로딩
    m_pAIScene = m_Importer.ReadFile(pModelFilePath, iFlag);
    if (nullptr == m_pAIScene)
        return E_FAIL;

    // 향후: aiScene에서 메시, 재질, 애니메이션 데이터 추출
    return S_OK;
}

HRESULT CModel::Initialize(void* pArg)
{
    return S_OK;
}

CModel* CModel::Create(ID3D11Device* pDevice, ID3D11DeviceContext* pContext, const _char* pModelFilePath)
{
    CModel* pInstance = new CModel(pDevice, pContext);

    if (FAILED(pInstance->Initialize_Prototype(pModelFilePath)))
    {
        MSG_BOX("Failed to Created : CModel");
        Safe_Release(pInstance);
    }

    return pInstance;
}

CComponent* CModel::Clone(void* pArg)
{
    CModel* pInstance = new CModel(*this);

    if (FAILED(pInstance->Initialize(pArg)))
    {
        MSG_BOX("Failed to Cloned : CModel");
        Safe_Release(pInstance);
    }

    return pInstance;
}

void CModel::Free()
{
    __super::Free();
    // Importer 소멸자가 aiScene을 자동 해제
    // → m_pAIScene을 직접 해제하면 안 됨!
}
```

### 10.4 핵심 분석

**`const _char*` 파일 경로**: Assimp은 멀티바이트(`char`) 경로를 사용. 기존 엔진의 `_tchar`(wchar_t)와 다르므로 주의.

**`Importer`와 `aiScene`의 수명 관계**:
```
m_Importer.ReadFile()
    → aiScene* 반환 (m_pAIScene에 저장)
    → aiScene의 메모리는 Importer가 소유!
    → Importer 소멸 시 aiScene도 자동 해제
    → 따라서 Free()에서 m_pAIScene을 직접 delete하면 안 됨
```

**복사 생성자의 한계**: 현재 Prototype에서 `m_pAIScene`과 `m_Importer`를 복사하지 않아 Clone이 실질적으로 빈 모델을 생성한다. 향후 메시 데이터를 별도 자료구조(정점/인덱스 버퍼)로 추출하면 이 문제가 해결됨.

## 12. CShader::Bind_RawValue — 복습

8일차에서 이미 구현한 메서드이지만, 9일차에서 **조명 데이터 바인딩의 핵심 수단**으로 활용되므로 사용 패턴을 복습한다.

### 12.1 기존 Bind 메서드 비교

| 메서드 | 대상 | HLSL 타입 | 내부 호출 |
|--------|------|-----------|-----------|
| `Bind_Matrix` | 4x4 행렬 | `float4x4` | `AsMatrix()->SetMatrix()` |
| `Bind_SRV` | 텍스처 | `texture2D` | `AsShaderResource()->SetResource()` |
| `Bind_RawValue` | 임의 데이터 | `vector`, `float`, etc. | `SetRawValue()` |

### 12.2 9일차에서의 사용

```cpp
// 카메라 위치 (float4 = 16바이트)
m_pShaderCom->Bind_RawValue("g_vCamPosition", m_pGameInstance->Get_CamPosition(), sizeof(_float4));

// 광원 방향 (float4 = 16바이트)
m_pShaderCom->Bind_RawValue("g_vLightDir", &pLightDesc->vDirection, sizeof(_float4));

// 광원 확산광 색상 (float4 = 16바이트)
m_pShaderCom->Bind_RawValue("g_vLightDiffuse", &pLightDesc->vDiffuse, sizeof(_float4));
```

**공통 패턴**: HLSL 변수 이름(문자열) + 데이터 포인터 + 바이트 크기. 이 세 가지만 맞추면 어떤 타입이든 전달 가능.

## 13. 수업 코드와 내 프로젝트의 차이점

수업 원본 코드와 내 프로젝트에는 이전 수업에서 변경한 부분들이 있다. 9일차 적용 시 주의할 차이점:

### 13.1 입력 시스템

| | 수업 코드 (9일차) | 내 프로젝트 |
|---|---|---|
| 방식 | DirectInput8 | Raw Input |
| 열거형 | `DIMB`, `DIMM` | `MOUSEBTN`, `MOUSEAXIS` |
| 키 식별 | `DIK_*` 스캔코드 | Virtual Key Code ('W' 등) |
| 헤더 | `<dinput.h>` 포함 | `<dinput.h>` 불필요 |

**주의**: 9일차의 `Engine_Enum.h`에서 `DIMB`/`DIMM`으로 표기되어 있지만, 내 프로젝트는 `MOUSEBTN`/`MOUSEAXIS`를 사용한다. 광원 시스템 적용 시 `LIGHT` 열거형만 추가하고, 입력 관련 열거형은 그대로 유지해야 한다.

### 13.2 Transform 시스템

| | 수업 코드 (9일차) | 내 프로젝트 |
|---|---|---|
| CTransform | `final` (구체 클래스) | `abstract` (추상 기반) |
| 파생 클래스 | 없음 | `CTransform_3D`, `CTransform_2D` |
| TRANSFORMTYPE | 없음 | GAMEOBJECT_DESC에 포함 |
| 생성 | `CTransform::Create()` 직접 | `eTransformType`에 따라 분기 |

**주의**: 9일차의 `GameObject.cpp`에서 `CTransform::Create()`를 직접 호출하지만, 내 프로젝트는 `CTransform_3D::Create()` / `CTransform_2D::Create()`로 분기한다. 이 차이는 광원 시스템 적용에 영향 없음.

9일차에서 가져올 것 (Light + Model):
